# Tic Tac Toe - Betclic Flutter Technical Test

A local Tic Tac Toe game built with Flutter.

The app lets a human player play a 3x3 game against a CPU opponent. It includes a home screen, in-game screen, settings, persisted scores, and three CPU difficulty levels.

## Features

- Human vs CPU local game
- 3x3 board
- Difficulty: easy, medium, hard
- Home settings: difficulty, theme, score reset
- In-game settings: theme and score reset; difficulty is locked for the current game
- Persisted scoreboard with `SharedPreferences`
- Light, dark, and system theme modes

## Architecture

The project follows Clean Architecture with dependencies pointing inward.

```text
lib/
├── core/
│   ├── di/
│   ├── router/
│   ├── storage/
│   └── theme/
└── features/
    └── game/
        ├── domain/
        │   ├── entities/
        │   ├── repositories/
        │   ├── services/
        │   └── usecases/
        ├── data/
        │   ├── datasources/
        │   ├── models/
        │   └── repositories/
        └── presentation/
            ├── controllers/
            ├── pages/
            └── widgets/
```

### Domain

Pure Dart. No Flutter, Riverpod, storage, routing, or DTO imports.

It owns the business concepts: `Board`, `GameSession`, `GameSettings`, `Scoreboard`, `GameRules`, CPU strategies, and use cases.

### Data

Implements domain repository contracts. It maps domain entities through JSON DTOs generated by `json_serializable` and stores values with an abstract `KeyValueStorage`.

### Presentation

Flutter UI and Riverpod controllers. It depends on domain use cases, not on concrete data repositories. Concrete dependencies are wired in `core/di`.

## CPU Strategies

- Easy: random legal move
- Medium: win, block, center, corners, sides
- Hard: minimax, deterministic and tested

## Architecture Guards

`import_lint` prevents common boundary violations:

- domain cannot import data, presentation, Flutter, Riverpod, GoRouter, or SharedPreferences
- data cannot import presentation
- presentation cannot import data

## Interview Notes

### `context.mounted` vs `mounted`

Use `context.mounted` after an async gap when the code holds a `BuildContext`. Use `mounted` inside a `State` object. In this app, navigation is synchronous from UI callbacks, and async work is handled in Riverpod controllers, so widget lifecycle checks are rarely needed.

### Why feature-first?

The main business area is the game. Keeping domain, data, and presentation under `features/game` makes the feature easy to move, test, and reason about while preserving Clean Architecture boundaries.

### How to prevent feature coupling?

Feature boundaries are enforced by import rules. A future feature should depend on shared `core` contracts or explicit public APIs, not on another feature's internals.

### How is winner detection tested?

`GameRules` is tested with unit tests. It is pure domain logic, so the tests cover rows, columns, diagonals, draw, and ongoing games without Flutter.

### Why Riverpod instead of BLoC?

Riverpod gives dependency injection and state management with less ceremony for this scope. The app keeps orchestration in controllers and business rules in use cases, so switching presentation state management would not affect the domain.

### Why not `flutter_hooks`?

The app does not need local hook-managed lifecycle or animation state. Adding hooks would not solve a real problem here.

### Generated Code Policy

`*.freezed.dart` and `*.g.dart` are ignored by Git. They are reproducible build outputs and must be regenerated with `dart run build_runner build`.
